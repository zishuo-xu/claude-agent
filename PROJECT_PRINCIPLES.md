# Project Principles

这个项目的目标是 **mini-claude**：一个参考 Claude Code 架构思想的轻量级工程化 agent。

它不是玩具 demo，也不是完整复刻 Claude Code。它要在较小代码量里保留 Claude Code 的核心架构边界，让学习者能理解 agent 如何工作，也让项目具备持续演进、测试和维护的基础。

Claude Code 对初学者来说结构完整但复杂。本项目会把其中值得学习的设计拆成轻量级工程化版本：能运行、能测试、能扩展，但不照搬生产级复杂度。

## 工作原则

1. **mini-claude 优先**
   每个实现都应服务于构建一个轻量级工程化的 Claude-style agent。它要帮助理解 Claude Code / coding agent 的核心机制，也要保持真实工程边界，而不是堆功能或做一次性 demo。

2. **及时写文档**
   所有重要实现、结构变化、设计取舍，都要及时更新文档，方便学习者和其他 AI 工具理解项目进展。
   每次操作后都要检查相关文档是否需要同步更新，保证文档反映当前真实状态，而不是滞后于代码。

3. **轻量级工程化**
   可以简化 Claude Code 的生产复杂度，但不能牺牲基本工程边界。核心能力要有清晰模块、测试、文档和可演进结构。

4. **架构对齐 Claude**
   架构与实现方案设计应优先参考 Claude Code 的设计。可以为了学习目的简化实现细节，但核心架构分层、概念边界和模块职责应尽量保持一致，避免做成一个只在功能表面相似、内部结构完全不同的项目。

5. **参考解读资料，但不照搬复杂度**
   除了 `references/Claude-Code-Source-Study`，后续设计也参考 CoreCoder 的 `article/` 系列解读。CoreCoder 文章主要用于帮助理解 Claude Code 的架构总览、Agent Loop、工具系统、上下文压缩、流式执行和多 Agent 等设计思想；本项目实现 mini-claude 所需的轻量工程化形态，不直接照搬生产级复杂度。

6. **保持简洁**
   不因为优化而盲目迭代，避免代码、文档和架构臃肿。每个新增功能、模块和文档段落都应服务于学习主线，能清楚说明为什么值得存在。能用清晰小实现表达的，就不要引入复杂抽象。
   不要轻易新增会造成代码层面或架构层面臃肿、不可维护的抽象。新增模块、边界或策略前，应先判断它是在降低既有复杂度，还是只是在制造新的概念负担。

7. **文档克制**
   文档要保持时效性，但不能靠重复堆内容来制造“完整感”。每份文档只承担一个职责；更新时优先修改最小必要位置，避免把同一信息同时写入 README、architecture、current features、roadmap 和 changelog。

8. **可运行优先**
   每一阶段都应该尽量保持项目可运行，避免只留下半成品架构。

9. **新增特性后推送 GitHub**
   每次新增特性并完成测试、文档同步和本地启动后，都要及时提交并推送到 GitHub，保证远端仓库反映当前项目进展。
   提交说明使用中文，并详细说明本次特性、影响范围、验证结果和必要的设计取舍。

10. **代码和文档互相对照**
   文档应明确指出某个 Claude Code 概念在本项目中对应哪个文件、类或函数。

11. **面向学习者解释**
   命名、注释、README 和学习路线都应帮助读者理解 agent 架构，同时保持工程表达准确，不把项目降级成玩具示例。

12. **适配层和核心逻辑分离**
   模型 API、终端 UI、外部工具协议都应该通过适配层接入。agent 的核心学习主线应保持清晰，不被某个供应商 SDK 细节淹没。

## 文档要求

新增或修改重要功能时，至少检查是否需要更新：

- `README.md`: 项目结构、运行方式、学习路线
- `PROJECT_PRINCIPLES.md`: 项目目标和长期原则
- `docs/architecture.md`: 当前架构和模块职责
- `docs/current-features.md`: 当前已经实现的功能
- `docs/roadmap.md`: 下一步工作、优先级和暂缓项
- `docs/versioning.md`: 版本规则和学习阶段
- `docs/learning-qa.md`: 独立学习沉淀，默认不参与日常上下文加载
- 相关模块内的简短注释或 docstring
- 未来可新增 `docs/` 目录，用专题文档解释复杂模块

文档时效性要求：

- 代码行为变了，更新功能文档
- 模块职责变了，更新架构文档
- 技术问题被解释清楚了，如有长期沉淀价值再更新独立学习文档
- 规划或优先级变了，更新 `docs/roadmap.md`
- 版本变化了，更新 `VERSION`、`CHANGELOG.md` 和 `docs/versioning.md`
- 运行方式、测试方式、配置方式变了，更新 README

文档减重要求：

- `README.md` 只做入口，不承载完整设计
- `docs/architecture.md` 只写稳定分层，不追逐每次实现细节
- `docs/current-features.md` 只写当前能力，不写历史
- `docs/roadmap.md` 只写方向和取舍，不复制 changelog
- `docs/versioning.md` 只写规则，不展开长历史
- `docs/learning-qa.md` 只收高价值技术解释，每条保持短，并从日常上下文入口中隔离
- `CHANGELOG.md` 记录版本变化，但每个版本避免重复已在功能文档中存在的内容

版本判断要求：

- 大特性通常提升 `MINOR`
- 小特性、修复通常提升 `PATCH`
- 纯文档变更通常不改版本，除非需要形成可追踪发布点
- 每次功能变化前先判断变更类型，再决定是否修改 `VERSION`

## 独立学习沉淀

学习发生在实现、代码审视和必要的沉淀文档中，不发生在最终产品交互里。

`docs/learning-qa.md` 是独立学习沉淀文档，不和 architecture、current features、roadmap 等工作文档耦合。日常设计和实现时默认不读取它，只有明确需要回看历史解释或学习记录时才打开。

这个文档不是聊天逐字稿，也不是所有问题都记录。普通聊天、项目规划、偏好约定、下一步安排、简单操作指令不进入 `docs/learning-qa.md`。高价值记录应尽量包含：

- 问题是什么
- 背后的原因
- 本项目如何处理
- 对应代码或文档入口

## 每次改动后的交付流程

每次修改代码、配置、依赖或运行行为后，默认执行：

1. 更新必要文档
2. 同步检查 `docs/architecture.md` 和 `docs/current-features.md`
3. 运行本地验证，例如 `python -m py_compile`、smoke test、CLI help
4. 启动当前 agent，让学习者可以立刻试用
5. 如果是新增特性，提交并推送到 GitHub

如果只修改文档，不需要启动或重启 agent；只需要做必要的轻量验证，例如检查关键文档内容或运行测试（视改动范围决定）。

如果某次改动不适合启动，例如缺少密钥、依赖安装失败、或启动会执行危险操作，需要明确说明原因。

如果新增 provider、工具、权限模式、上下文策略等重要能力，应同时说明：

- 它参考了 Claude Code 的哪个概念
- 本项目做了哪些简化
- 它在架构分层上如何与 Claude Code 保持对应
- 学习者应该从哪个文件开始读

## 当前主线

本项目当前围绕这条主线展开：

```text
用户目标 -> 模型推理 -> 工具调用 -> 权限判断 -> 本地执行 -> 结果回传 -> 下一轮推理
```

后续扩展也应围绕这条主线逐步展开，而不是一次性引入过多生产级概念。
