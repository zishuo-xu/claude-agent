# Stress Test Cases

这份文档保存可复用的长压力测试案例。它不记录项目设计，只提供下次验收时可直接复制的任务说明。

使用原则：

- 优先在临时目录下执行，避免污染项目文件。
- 每个案例都要覆盖“多轮上下文延续 + 工具调用 + 验证”。
- 测试后删除临时产物，除非需要保留失败现场。
- 如果只做验收，记录到 `docs/maintenance-log.md`，不升版本。
- 如果发现并修复用户可见问题，按 PATCH 记录。

## 案例 1：长代码任务

目标：验证 agent 能连续创建代码、修改同一文件、运行验证。

可直接发给 agent：

```text
做一次长压力测试。请在临时目录 tmp_stress_code 里完成：

1. 创建 calc.py，实现 add(a, b)。
2. 修改 calc.py，增加 subtract(a, b)。
3. 再修改 calc.py，增加 multiply(a, b)。
4. 再修改 calc.py，增加 divide(a, b)，要求除数为 0 时抛 ValueError。
5. 再修改 calc.py，加入 __main__ 自检，覆盖 add/subtract/multiply/divide。
6. 运行 python3 tmp_stress_code/calc.py 验证输出 ok。
7. 总结每一轮是否正确使用同一个文件、是否完成验证。

要求：每一步都通过工具实际创建或修改文件，不只口头说明。
```

通过标准：

- `tmp_stress_code/calc.py` 存在。
- 文件包含 `add`、`subtract`、`multiply`、`divide`。
- `python3 tmp_stress_code/calc.py` 输出 `ok`。
- 中途不会丢失前面已写入的函数。

## 案例 2：长写作任务

目标：验证创作内容、保存文件、继续追加、切换话题后再回到文件任务。

可直接发给 agent：

```text
做一次长压力测试。请在临时目录 tmp_stress_story 里完成：

1. 创建 story.md，写一个校园故事大纲，要求至少 5 个主要角色。
2. 继续追加第一章开头。
3. 暂时切换话题：解释当前项目架构，要求简短。
4. 再回到 story.md，继续追加第二章开头。
5. 读取 story.md，确认大纲、第一章、第二章都还在。
6. 总结是否出现误读项目文件、遗忘上文、覆盖旧内容的问题。
```

通过标准：

- `tmp_stress_story/story.md` 存在。
- 文件同时包含大纲、第一章、第二章。
- 切换项目问答后，回到明确文件路径仍能继续编辑该文件。
- 不会把项目 README 或架构文档误写进故事文件。

## 案例 3：上下文压缩压力

目标：验证长输出、工具结果和后续任务在上下文压力下不会互相覆盖。

可直接发给 agent：

```text
做一次上下文压力测试。请在临时目录 tmp_stress_context 里完成：

1. 创建 notes.md，写入 20 条编号笔记，每条一句话。
2. 读取 notes.md 并总结前 5 条和后 5 条。
3. 追加 20 条新笔记，编号从 21 到 40。
4. 再读取 notes.md，确认 1-40 都存在。
5. 创建 summary.md，写一个不超过 10 行的总结。
6. 运行 wc -l tmp_stress_context/notes.md tmp_stress_context/summary.md。
7. 总结是否发生工具结果刷屏、旧内容丢失或 summary 污染当前任务。
```

通过标准：

- `notes.md` 包含 1-40 的连续编号。
- `summary.md` 不超过 10 行。
- 工具结果展示应保持可读，不应大段刷屏。

## 案例 4：权限和取消

目标：验证写入权限确认、取消任务和取消后继续不会恢复旧任务。

可直接发给 agent：

```text
做一次权限和取消压力测试。请在 plan 权限模式下完成：

1. 请求创建 tmp_stress_cancel/a.txt，内容是 alpha。
2. 当需要权限时，我会同意。
3. 再请求创建 tmp_stress_cancel/b.txt，但在你询问缺失信息或确认时，我会说“算了”。
4. 之后我再说“继续”，你应该先澄清，而不是恢复 b.txt 的创建。
5. 最后列出 tmp_stress_cancel，确认只有 a.txt。
```

通过标准：

- `a.txt` 被创建。
- 取消后不会创建 `b.txt`。
- 取消后“继续”不会恢复已取消任务。

## 案例 5：项目问答输出收敛

目标：验证项目问答只读最小文档，不输出整份设计。

可直接发给 agent：

```text
做一次项目问答压力测试：

1. 问“当前项目架构是什么”，要求回答不超过 6 条。
2. 继续问“怎么启动”，要求只给命令。
3. 继续问“下一步做什么，为什么”，要求说明理由和作用。
4. 总结每一步读了哪些文件，是否出现无关长篇输出。
```

通过标准：

- 架构回答不展开全部历史。
- 启动回答只给必要命令。
- 下一步计划包含理由和作用。
- 工具读取应优先命中文档入口，而不是先扫全项目。
