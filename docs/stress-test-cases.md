# Stress Test Cases

这份文档保存可复用的长压力测试案例。它不记录项目设计，只提供下次验收时可直接复制的任务说明。

使用原则：

- 优先在临时目录下执行，避免污染项目文件。
- 每个案例都要覆盖“多轮上下文延续 + 工具调用 + 验证”。
- 每次执行后必须输出验证结果，不能只说“完成”。
- 测试后删除临时产物，除非需要保留失败现场。
- 如果只做验收，记录到 `docs/maintenance-log.md`，不升版本。
- 如果发现并修复用户可见问题，按 PATCH 记录。


## 案例 1：长代码任务

目标：验证 agent 能连续创建代码、修改同一文件、运行验证。

可直接发给 agent：

```text
做一次长压力测试。请在临时目录 tmp_stress_code 里完成：

1. 创建 calc.py，实现 add(a, b)。
2. 修改 calc.py，增加 subtract(a, b)。
3. 再修改 calc.py，增加 multiply(a, b)。
4. 再修改 calc.py，增加 divide(a, b)，要求除数为 0 时抛 ValueError。
5. 再修改 calc.py，加入 __main__ 自检，覆盖 add/subtract/multiply/divide。
6. 运行 python3 tmp_stress_code/calc.py 验证输出 ok。
7. 总结每一轮是否正确使用同一个文件、是否完成验证。

要求：每一步都通过工具实际创建或修改文件，不只口头说明。
```

通过标准：

- `tmp_stress_code/calc.py` 存在。
- 文件包含 `add`、`subtract`、`multiply`、`divide`。
- `python3 tmp_stress_code/calc.py` 输出 `ok`。
- 中途不会丢失前面已写入的函数。

## 案例 2：长写作任务

目标：验证创作内容、保存文件、继续追加、切换话题后再回到文件任务。

可直接发给 agent：

```text
做一次长压力测试。请在临时目录 tmp_stress_story 里完成：

1. 创建 story.md，写一个校园故事大纲，要求至少 5 个主要角色。
2. 继续追加第一章开头。
3. 暂时切换话题：解释当前项目架构，要求简短。
4. 再回到 story.md，继续追加第二章开头。
5. 读取 story.md，确认大纲、第一章、第二章都还在。
6. 总结是否出现误读项目文件、遗忘上文、覆盖旧内容的问题。
```

通过标准：

- `tmp_stress_story/story.md` 存在。
- 文件同时包含大纲、第一章、第二章。
- 切换项目问答后，回到明确文件路径仍能继续编辑该文件。
- 不会把项目 README 或架构文档误写进故事文件。

## 案例 3：上下文压缩压力

目标：验证长输出、工具结果和后续任务在上下文压力下不会互相覆盖。

可直接发给 agent：

```text
做一次上下文压力测试。请在临时目录 tmp_stress_context 里完成：

1. 创建 notes.md，写入 20 条编号笔记，每条一句话。
2. 读取 notes.md 并总结前 5 条和后 5 条。
3. 追加 20 条新笔记，编号从 21 到 40。
4. 再读取 notes.md，确认 1-40 都存在。
5. 创建 summary.md，写一个不超过 10 行的总结。
6. 运行 wc -l tmp_stress_context/notes.md tmp_stress_context/summary.md。
7. 总结是否发生工具结果刷屏、旧内容丢失或 summary 污染当前任务。
```

通过标准：

- `notes.md` 包含 1-40 的连续编号。
- `summary.md` 不超过 10 行。
- 工具结果展示应保持可读，不应大段刷屏。

## 案例 4：权限和取消

目标：验证写入权限确认、取消任务和取消后继续不会恢复旧任务。

可直接发给 agent：

```text
做一次权限和取消压力测试。请在 plan 权限模式下完成：

1. 请求创建 tmp_stress_cancel/a.txt，内容是 alpha。
2. 当需要权限时，我会同意。
3. 再请求创建 tmp_stress_cancel/b.txt，但在你询问缺失信息或确认时，我会说“算了”。
4. 之后我再说“继续”，你应该先澄清，而不是恢复 b.txt 的创建。
5. 最后列出 tmp_stress_cancel，确认只有 a.txt。
```

通过标准：

- `a.txt` 被创建。
- 取消后不会创建 `b.txt`。
- 取消后“继续”不会恢复已取消任务。

## 案例 5：项目问答输出收敛

目标：验证项目问答只读最小文档，不输出整份设计。

可直接发给 agent：

```text
做一次项目问答压力测试：

1. 问“当前项目架构是什么”，要求回答不超过 6 条。
2. 继续问“怎么启动”，要求只给命令。
3. 继续问“下一步做什么，为什么”，要求说明理由和作用。
4. 总结每一步读了哪些文件，是否出现无关长篇输出。
```

通过标准：

- 架构回答不展开全部历史。
- 启动回答只给必要命令。
- 下一步计划包含理由和作用。
- 工具读取应优先命中文档入口，而不是先扫全项目。

## 高压案例 6：小型项目从 0 到验证

目标：验证 agent 能连续创建多文件项目、跨文件修改、修复错误并运行测试。

可直接发给 agent：

```text
做一次高压长任务测试。请在临时目录 tmp_stress_project 里完成一个小型 Python 项目：

1. 创建包目录 textstats/，包含 __init__.py、core.py、cli.py。
2. core.py 实现 count_words、count_lines、top_words 三个函数。
3. cli.py 实现命令行入口：读取一个文本文件，输出行数、词数、top 3 单词。
4. 创建 tests/test_core.py，至少覆盖空文本、多行文本、重复单词。
5. 创建 sample.txt，内容至少 20 行。
6. 运行 pytest tests/test_core.py。
7. 故意修改一个测试期望，让测试失败一次，观察失败信息。
8. 再修复测试或代码，让测试重新通过。
9. 运行 python -m textstats.cli sample.txt，确认 CLI 输出合理。
10. 最后总结创建了哪些文件、失败如何恢复、最终验证命令是什么。

要求：全过程必须实际创建和修改文件，不能只给代码块。
```

通过标准：

- 至少创建 5 个文件。
- 至少经历一次测试失败和一次修复。
- 最终 pytest 通过。
- CLI 可以运行并输出统计结果。
- 修复过程中不丢失前面已创建文件。

## 高压案例 7：多主题切换和回到任务

目标：验证长对话中多次切换主题后，明确回到文件任务时仍能继续。

可直接发给 agent：

```text
做一次多主题切换压力测试。请在临时目录 tmp_stress_switch 里完成：

1. 创建 novel.md，写一个故事大纲，至少 6 个角色、3 条主线。
2. 继续追加第一章。
3. 切换话题：解释当前项目 Agent Loop，不超过 5 条。
4. 回到 novel.md，追加第二章。
5. 再切换话题：说明怎么启动当前项目，只给命令。
6. 回到 novel.md，追加第三章。
7. 创建 characters.md，整理 6 个角色关系。
8. 读取 novel.md 和 characters.md，确认章节和角色关系都存在。
9. 总结是否出现误读项目文档、忘记目标文件、覆盖旧章节的问题。
```

通过标准：

- `novel.md` 包含大纲、第一章、第二章、第三章。
- `characters.md` 包含 6 个角色关系。
- 项目问答不污染创作文件。
- 回到文件任务必须明确使用目标路径。

## 高压案例 8：大文件追加和摘要边界

目标：验证工具结果预算、上下文压缩和摘要边界。

可直接发给 agent：

```text
做一次大文件上下文压力测试。请在临时目录 tmp_stress_large 里完成：

1. 创建 log.md，写入 100 条编号记录，每条包含 id、category、message。
2. 读取 log.md，只总结 category 分布，不要把 100 条全部输出到终端。
3. 追加 101-200 条记录，category 要延续前面的几类。
4. 再读取 log.md，确认编号 1、50、100、150、200 都存在。
5. 创建 report.md，写一个 12 行以内的摘要报告。
6. 运行 wc -l tmp_stress_large/log.md tmp_stress_large/report.md。
7. 总结是否发生工具结果刷屏、摘要覆盖原始内容、编号断裂的问题。
```

通过标准：

- `log.md` 至少 200 行或 200 条记录。
- `report.md` 不超过 12 行。
- 终端展示不能刷出完整大文件。
- 编号关键点 1、50、100、150、200 都存在。

## 高压案例 9：错误恢复和权限边界

目标：验证命令失败、修复、权限确认和取消不会打乱任务状态。

可直接发给 agent：

```text
做一次错误恢复压力测试。请在临时目录 tmp_stress_recovery 里完成：

1. 创建 app.py，里面故意包含一个语法错误。
2. 运行 python3 tmp_stress_recovery/app.py，确认失败。
3. 根据错误信息修复 app.py。
4. 再运行，确认输出 ok。
5. 创建 config.json，写入 {"mode": "test"}。
6. 请求删除 config.json，但随后取消这个删除动作。
7. 再说“继续”，你应该澄清当前目标，而不是继续删除。
8. 最后列出 tmp_stress_recovery，确认 app.py 和 config.json 都存在。
```

通过标准：

- 至少经历一次命令失败和一次修复成功。
- 取消删除后文件仍存在。
- “继续”不会恢复已取消的破坏性动作。

## 高压案例 10：综合验收

目标：把代码、文档、项目问答、长文件和验证串成一个综合压力测试。

可直接发给 agent：

```text
做一次综合高压测试。请在临时目录 tmp_stress_full 里完成：

1. 创建 README.md，说明这是一个临时 notes 工具。
2. 创建 notes_tool.py，实现 add_note、list_notes、search_notes。
3. 创建 notes.json，初始化为空列表。
4. 创建 test_notes_tool.py，覆盖新增、列表、搜索。
5. 运行 pytest tmp_stress_full/test_notes_tool.py。
6. 如果失败，修复后重跑，直到通过。
7. 创建 usage.md，写 5 条使用示例。
8. 切换话题：简短解释当前 mini-claude 和 Claude 在上下文管理上的差异。
9. 回到 tmp_stress_full/usage.md，追加一节“验收结果”。
10. 运行 ls tmp_stress_full 和 wc -l tmp_stress_full/*.md。
11. 总结工具链是否完成：创建、修改、测试、切换话题、回到文件、最终验证。

要求：不要把项目源码改到 tmp_stress_full 之外。测试结束后说明临时目录可以删除。
```

通过标准：

- 至少创建 5 个文件。
- 至少运行一次 pytest。
- 切换项目问答后能回到 `usage.md`。
- 不修改临时目录之外的项目源码。
- 最终总结包含失败点、修复点和验证命令。
